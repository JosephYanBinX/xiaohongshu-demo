# Skill 定义：汇报 Demo 生成助手

> 基于 2026-03 小红书 + 点点AI demo 项目的经验提炼

---

## 这个 Skill 做什么

给定一个产品概念，快速生成一个可在浏览器直接运行的交互原型（HTML/CSS/JS），用于向上汇报或团队评审。

输入：口述的产品想法
输出：可运行的 demo + 部署链接

---

## 第一步：提前问清楚的问题（必须，不能省）

每次开始前，先把这几个问题问完再动手，否则后期大概率返工：

| 问题 | 为什么重要 |
|------|-----------|
| 演示给谁看？（老板 / 投资人 / 团队内部） | 决定细节深度和 mock 数据的真实程度 |
| 核心要演示哪条路径？（用户侧 / 创作者侧 / 两者都要） | 决定页面数量和优先级 |
| 有没有「必须能点」的关键交互？ | 这些交互要单独处理，不能只是静态页面 |
| 有没有参考的设计稿或截图？ | 有的话直接对照，节省大量来回确认 |
| 预计多久要用？ | 决定做到什么程度，别过度设计 |

## 核心工作方式（从这次提炼的）

### 文件结构：先定好，不要改

```
project/
├── index.html      # 所有页面的 HTML 骨架
├── css/style.css   # 所有样式
└── js/
    ├── data.js     # 所有 mock 数据（和渲染逻辑完全分离）
    └── app.js      # 导航系统 + 页面渲染 + 交互逻辑
```

**data.js 和 app.js 分离是关键。** 后期改数据（换文案、加字段、调 mock 内容）完全不需要碰渲染逻辑。

### 构建顺序：骨架优先

1. **先搭导航系统**：定义所有页面 ID，让所有页面可以互相跳转，哪怕内容还是空的
2. **再填数据结构**：在 data.js 里把所有实体（用户、内容、功能项等）定义清楚，mock 数据要认真写，假数据会让 demo 失去说服力
3. **逐页实现内容**：按用户路径的先后顺序，一页一页做完
4. **最后处理细节交互**：动画、状态切换、边界情况

### Mock 数据的原则

- **数量要够**：列表类内容至少 4-6 条，不能只有 2 条
- **内容要真实**：用真实的文案、真实的数字，不要「XXX」占位符
- **覆盖不同状态**：比如「已安装」和「未安装」的 Skill 都要有，让演示路径更灵活

### 交互设计的确认节点

**凡是有「用户输入 → AI 响应」的流程，必须设计确认节点。** 不能直接跳过用户的判断直接给结果。

本次经验：菜单助手的「草稿 → 用户确认 → 购物清单」是后来补的，应该在第一轮就设计进去。

节点设计模式：
```
AI 输出中间结果
↓
明确提示用户「确认后继续」（文字说清楚怎么操作）
↓
输入框激活（视觉上有变化，让用户知道可以输入了）
↓
用户输入确认
↓
AI 输出最终结果
```

## 容易返工的地方（避坑清单）

| 坑 | 原因 | 避免方式 |
|----|------|---------|
| 底部操作栏交互改了好几轮 | 没提前问清楚「已安装」状态下应该显示什么 | 第一步就把所有状态穷举出来确认 |
| 重复 ID 导致弹层打不开 | 多页面同时在 DOM 里，getElementById 会命中隐藏页面的元素 | 所有 ID 查询改为在当前页面作用域内查找 |
| 对话动画是后来加的 | 一开始没想到 demo 里的「感受」也很重要 | 凡是对话流，默认加逐条出现动画 |
| Markdown 渲染漏了某些页面 | 只在部分地方加了 md-content 类 | 所有 AI 回复的气泡统一用同一个函数渲染 |

## 部署（标准流程）

```bash
# 首次
git init && git add . && git commit -m "init"
git remote add origin https://github.com/用户名/仓库名.git
git push -u origin main

# 后续更新
git add . && git commit -m "update" && git push
```

Vercel 连接 GitHub 仓库后，每次 push 自动重新部署，链接不变。

国内访问用 Gitee Pages（gitee.com），流程相同。

## 下次启动一个新 Demo 时的 checklist

- [ ] 问清楚上方「第一步」的 5 个问题
- [ ] 确认文件结构和技术选型（纯 HTML/CSS/JS，不引入框架）
- [ ] 把所有页面 ID 和跳转关系列出来，双方确认
- [ ] 把核心实体和数据字段定义好，再开始写代码
- [ ] 把「必须能点」的交互节点标出来，优先实现
- [ ] 有对话流的地方，默认加确认节点和动画
